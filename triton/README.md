# Triton + TensorRT-LLM (second inference backend)

> Phase 2.0's headline lesson: the gateway seam holds under a real
> backend swap. The agent's `model="qwen-chat-trt"` request goes through
> LiteLLM to **Triton serving a TensorRT-LLM-compiled engine**, with
> *no code changes*. Different metric vocabulary (`nv_inference_*` vs
> `vllm:*`), different perf profile, same agent.

## What this is

[Triton Inference Server](https://github.com/triton-inference-server/server)
is NVIDIA's general-purpose model server. We use it with the
[TensorRT-LLM backend](https://github.com/triton-inference-server/tensorrtllm_backend),
which serves a compiled `.engine` file produced by TensorRT-LLM's build
tools. The result is a binary optimised for one specific GPU
architecture — for our Ada (RTX 4060), that's `SM 8.9`.

Two consequences worth internalising:

1. **The engine is GPU-specific.** A `.engine` compiled on a 4060 will
   not run on an A100, an H100, or even a 4080. Rebuild on every new GPU.
2. **The engine is large.** ~3 GB for Qwen2.5-3B at int4. Committing
   that to git would be silly; it's regenerated by
   [`scripts/build-trt-engine.sh`](../scripts/build-trt-engine.sh).

## Why it's here

Phase 1 set up vLLM behind LiteLLM. Phase 2.0 demonstrates that "behind
LiteLLM" is a real seam: another inference engine, with a genuinely
different metric surface and HTTP shape, plugs in by editing
`litellm/config.yaml` and adding one compose service. The agent app
never knows.

The plan documents this as *the* headline Phase-2 lesson. Phase 2.1–2.3
were observability scaffolding so that when we swap engines, we can
*see* the swap on the dashboards (TRT-LLM panels vs vLLM panels) and in
the logs.

## How it's wired

```
   agent  ─┬─ "qwen-chat"     ──┐
            │                    ▼
            │           ┌─────────────────┐         ┌────────────────┐
            └─ "qwen-chat-trt" ─│   LiteLLM       │ ─ http ─▶ │   vllm-engine  │
                        │   gateway       │           └────────────────┘
                        │   :4000         │ ─ http ─▶ ┌────────────────┐
                        └─────────────────┘           │ triton-server  │
                                                       │   :8002 (HTTP) │
                                                       │   :8004 (metr) │
                                                       └────────────────┘
                                                          ▲
                                                    /v2/models/ensemble/generate
```

LiteLLM's native `triton/` provider handles the wire-format translation —
it sends `text_input` / `text_output` to Triton's standard
`/v2/models/<name>/generate` API and serialises back into OpenAI's chat
completion shape. The agent gets the same JSON, regardless of which
backend serviced it.

## Disk / compute budget

| Step | Disk | Time on Ada |
| ---- | ---- | ----------- |
| Pull `nvcr.io/nvidia/tritonserver:25.04-trtllm-python-py3` | ~25 GB unpacked | one-off via `scripts/pull-phase2-images.sh` |
| Download fp16 Qwen2.5-3B-Instruct weights (HF cache) | ~6 GB | ~2-3 min |
| Convert HF → TRT-LLM checkpoint | ~5 GB intermediate (in container `/tmp`) | ~3-5 min |
| Build `.engine` | ~3 GB final | ~7-10 min |
| Triton model_repository total | ~3.5 GB on host | — |

GPU: **build needs the GPU exclusive** — stop vLLM first
(`docker compose stop vllm-engine`) before running the build script.

## End-to-end workflow

```bash
# 1. (one-off, fresh clone) make sure the Triton image is pulled
./scripts/pull-phase2-images.sh

# 2. free the GPU
docker compose stop vllm-engine

# 3. compile the engine + populate the model repository (~10-15 min)
./scripts/build-trt-engine.sh

# 4. bring Triton up under its profile
docker compose --profile triton up -d triton-server

# 5. smoke test through the gateway
curl -s :4000/v1/chat/completions \
  -H "Authorization: Bearer $(grep ^LITELLM_MASTER_KEY .env | cut -d= -f2)" \
  -H 'Content-Type: application/json' \
  -d '{"model":"qwen-chat-trt","messages":[{"role":"user","content":"say hi"}]}' \
  | jq '.choices[0].message.content'
# → "Hello! How can I help you today?"

# 6. (swap back) take Triton down, bring vLLM back
docker compose --profile triton down triton-server
docker compose up -d vllm-engine
```

## Configuration files

| Where | What |
| ----- | ---- |
| [model_repository/](model_repository/) | Generated by `build-trt-engine.sh` from the `inflight_batcher_llm` template in the Triton image. Five sub-models: `preprocessing`, `tensorrt_llm`, `postprocessing`, `ensemble` (the model LiteLLM hits), and `tensorrt_llm_bls`. **Gitignored**; rebuild on the host. |
| [engines/](engines/) | Raw `.engine` build artefacts before they're copied into the model repository. **Gitignored**. |
| `docker-compose.yaml` (`triton-server`) | Service definition under `profiles: [triton]` — opt-in. Mounts `model_repository/` read-only. |
| `litellm/config.yaml` (`qwen-chat-trt`) | Routing entry using LiteLLM's native `triton/` provider. |
| `prometheus/prometheus.yml` (`triton` job) | Scrapes `triton-server:8004`. Goes DOWN when the profile is off — expected. |
| `grafana/dashboards/03-trt-llm.json` | Per-backend dashboard with `nv_inference_*` panels. |

## Switching the app to Triton without code changes

Two ways, both demonstrate the "agent doesn't know" lesson:

**(a) Edit the agent's model name.** In `app/agent.py`, change
`model="qwen-chat"` to `model="qwen-chat-trt"`, restart the app. One
line, one restart.

**(b) Router alias — the slicker version.** Uncomment the
`router_settings.model_group_alias` block in `litellm/config.yaml`:

```yaml
router_settings:
  model_group_alias:
    qwen-chat: ["qwen-chat-trt", "qwen-chat"]
```

Now every `qwen-chat` request prefers `qwen-chat-trt` (with `qwen-chat`
as fallback). The agent still asks for `qwen-chat`; LiteLLM transparently
routes to Triton. `docker compose restart litellm` to reload.

## Smoke tests

```bash
# Triton ready?
curl -s http://localhost:8002/v2/health/ready
# → no body, 200

# Model loaded?
curl -s http://localhost:8002/v2/models/ensemble/ready
# → no body, 200

# Triton's own /metrics
curl -s http://localhost:8004/metrics | grep -E '^nv_inference_(count|request_success)'
```

Through the gateway:

```bash
LITELLM_KEY=$(grep ^LITELLM_MASTER_KEY .env | cut -d= -f2)

# Single request via LiteLLM → Triton
curl -s :4000/v1/chat/completions \
  -H "Authorization: Bearer $LITELLM_KEY" \
  -H 'Content-Type: application/json' \
  -d '{"model":"qwen-chat-trt","messages":[{"role":"user","content":"say hi"}]}' \
  | jq '{ok:(.choices|length), content:.choices[0].message.content}'

# Drive load against TRT-LLM specifically (Phase 2.0 dashboards light up)
./scripts/load.sh --model qwen-chat-trt short
```

## Where to look when it breaks

| Symptom | Likely cause | Fix |
| ------- | ------------ | --- |
| `build-trt-engine.sh` fails at `convert_checkpoint.py` with `path not found` | TRT-LLM examples moved between minor versions | The script tries `models/core/qwen` then `qwen`; if both fail, `docker run --rm $TRT_IMAGE find /app/tensorrt_llm/examples -name convert_checkpoint.py` to find the right path |
| `trtllm-build` OOMs | Host RAM < 8 GB free during compile | Bump host memory, or lower `--max_batch_size` and `--max_input_len` |
| `triton-server` starts but `/v2/health/ready` returns 503 | One of the five sub-models failed to load. Common culprit: `tensorrt_llm/1/` doesn't have the `.engine` file | `docker compose logs triton-server` — the failing model is named in the error. Re-run `build-trt-engine.sh` |
| `triton-server` healthy but LiteLLM 500s on `qwen-chat-trt` | Wrong model name in `triton/<name>` — must match a model in `model_repository/` | `curl :8002/v2/models` to list loaded models; the LiteLLM config says `triton/ensemble` |
| Mean latency on the TRT-LLM dashboard is *higher* than vLLM | TRT-LLM batching tuning matters more than vLLM's: low `kv_cache_free_gpu_mem_fraction`, paged-FMHA disabled, etc. | Re-build with `--use_paged_context_fmha enable` and check the engine config.json |
| Engine compile finishes but Triton refuses to load it with `engine plan mismatch` | Engine was built for a different GPU architecture than the host | The `.engine` is SM-specific. Rebuild on the host you'll run it on |
| Both engines fail with `out of memory` after a swap | A previous container is still holding GPU memory | `nvidia-smi` to confirm; `docker compose down vllm-engine triton-server` and start fresh |

## Honesty about the difficulty

This is the most fragile phase of the lab. TRT-LLM's tooling moves fast,
the example directory layout changes between minor releases, and the
quantization story for Qwen has been in flux. If `build-trt-engine.sh`
breaks, the failure mode is almost always in `convert_checkpoint.py` or
the `fill_template.py` invocations — both of which depend on internal
file layouts inside `nvcr.io/nvidia/tritonserver:25.04-trtllm-python-py3`.

The plan acknowledges this in §2.0: pin the Triton image (we do), and
treat the build as a one-off chore the user does outside compose. If
the chore fails on a future image bump, this README's troubleshooting
matrix should get you most of the way back.

## Known limitations of the current build

Phase 2.0 demonstrates the **gateway-swap lesson** end-to-end: app →
LiteLLM → Triton, with the agent unchanged. The quality of *generated
tokens* is a separate concern, and our current build is deliberately
on the cheap-and-cheerful end:

| Concern | Current state | Root cause | Fix path |
| ------- | ------------- | ---------- | -------- |
| Output reads as gibberish ("akenspacessly", random fragments) | Real | `convert_checkpoint.py --weight_only_precision int4` is **naive** int4 quant with no calibration; ~3B-param chat-tuned models lose most of their structure under this | Switch to real AWQ via TRT-LLM's `examples/quantization/quantize.py` (modelopt-based, ~30 min build, needs a small calibration dataset). Adds disk + time but produces vLLM-comparable output |
| Token-usage counters read 0/0/0 | Real | LiteLLM's `triton/` provider doesn't extract `prompt_tokens` / `completion_tokens` from Triton's `text_output` response | Same root-cause class as the Phase 2.2 token-usage limitation documented in `app/README.md` — upstream interop |
| Streaming with the default ensemble | Disabled | Our `tensorrt_llm` model is `decoupled_mode: false`; LiteLLM's `triton/` chat-completion path defaults to streaming for some configs | The agent's `ChatOpenAI` is pinned to `streaming=False` (see `app/agent.py`) so this can't drift. Real streaming support is on Phase 2.5's plate |
| Engine is GPU-architecture-specific | Real | `.engine` is compiled for SM 8.9 | Rebuild on each new GPU. Not portable, can't be cached in a registry the way a docker image can |

The "swap engines via litellm/config.yaml" lesson is intact — the agent
talks to whichever backend the gateway routes to without code changes.
Phase 2.0 is *demonstrably* wired; the *quality* of the served tokens
is the next thing to invest in if this lab grows beyond the seam demo.
